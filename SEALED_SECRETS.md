# Sealed secrets

This chart deliberately does **not** ship sealed secrets, because they're
cluster-bound: a SealedSecret is encrypted against one specific controller's
key, so committing one to the chart would only work for one deployment.

Instead, the chart's manifests reference secrets by name. You provision those
secrets out-of-band before running `helm install` ŌĆö either as plain `Secret`
manifests (fine for testing), sealed with `kubeseal` (production), or fetched
by an operator like External Secrets (Phase 2).

`scripts/generate-secrets.sh` in this repo automates the whole flow: it mints
every secret below and optionally seals each one against your cluster's
controller.

## Expected secrets

Each one is namespaced into a `cv-*` namespace by name (or by your
`namespacePrefix`). All keys are required unless marked optional.

### `portal-platform-secrets` in `<prefix>portal`

Security-hardening secrets. The portal runs `NODE_ENV=production` and fails
closed without them; the mcp-gateway 403s every `/mcp` request without the
webhook secret.

| Key | Value |
|---|---|
| `PORTAL_SECRET_KEY` | base64-encoded **exactly 32 bytes** (`openssl rand -base64 32`) ŌĆö AES-256 key that encrypts per-project Postgres passwords at rest. The portal refuses to boot without it. To rotate: set the new key plus `PORTAL_SECRET_KEY_OLD` and restart; drop the old once migrated. |
| `INTERNAL_WEBHOOK_SECRET` | random 32+ char string ŌĆö authenticates `/internal` callers (`X-Internal-Secret` header). Shared with the mcp-gateway for its per-request project-ownership check. |

## Generation flow

This repo ships `scripts/generate-secrets.sh` which:

1. Takes operator-supplied inputs as flags (`--smtp-uri`,
   `--cloudflare-credentials`).
2. Generates random per-deployment values (Postgres passwords, Hydra/Kratos
   system secrets) ŌĆö written to `./out/secrets.local.env` so you can
   regenerate `DSN_*` etc. without losing entropy (`--restore-env`).
3. Renders each `Secret` to `./out/secrets/plain/<name>.yaml`.
4. With `--seal`, pipes each through `kubeseal
   --controller-name=sealed-secrets-controller
   --controller-namespace=kube-system -o yaml` and writes
   `./out/secrets/sealed/<name>.sealed.yaml`.
5. You apply the sealed YAMLs (and may safely commit them to your own
   deployment repo ŌĆö only your cluster's controller can decrypt them).

The sealed YAMLs land in the right namespace because the secrets templates
hardcode them.

## Master-key backup

A SealedSecret is encrypted against the controller's master RSA key. If you
lose the key, the sealed yamls become unusable and every secret must be
re-sealed against a new key. Capture the controller key right after
installing the controller:

    kubectl get secrets -n kube-system -l sealedsecrets.bitnami.com/sealed-secrets-key \
      -o yaml > sealed-secrets-master.local.yaml

Store that file somewhere SAFE (encrypted backup, password manager export,
SOPS-encrypted ŌĆö never in plain git). To restore on a new cluster:

    kubectl apply -f sealed-secrets-master.local.yaml
    kubectl delete pod -n kube-system -l app.kubernetes.io/name=sealed-secrets

The controller picks up the restored key on restart and decrypts existing
sealed yamls without regeneration.

## When you can skip sealed-secrets

For a one-off test cluster, you can apply the secrets as plain `Secret`
manifests directly ŌĆö the workloads don't care how the data got there. Just
remember to delete the cluster before forgetting about it.
